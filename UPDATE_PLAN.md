# Algarve T3 — мини-база: итог проекта + план обновления

## 0. Что это
Интерактивная база подборки **T3-квартир/домов** в Algarve (Quarteira · Vilamoura · Almancil · Loulé), бюджет ≤1 000 000 €, энергокласс A/A+/B, постройка 2015+ или новостройка, желательно с гаражом/паркингом.
Итог — самодостаточный `algarve-t3-final.html`: карта, топ-подборки по критериям, полный список со score/rank, фото, описания, живые слайдеры весов.
**Текущий объём: 78 объектов** (60 idealista + 18 не-idealista), на 2026-05-30.

## 1. Жёсткие критерии (фильтр включения)
- Только **T3** (3 спальни; T3+1/таунхаус/вилла T3 — ок; T2/T4/T5 — нет).
- Цена **≤ 1 000 000 €** (для застроек — цена именно T3-юнита).
- Энергокласс **A+/A/B** (B- допустимо с пометкой).
- Год **≥ 2015** ИЛИ новостройка/off-plan (НЕ 2027+ дальний off-plan).
- Зоны: **Quarteira, Vilamoura, Almancil, Loulé** (НЕ Albufeira/Olhos de Água).
- Гараж/личный паркинг — приоритет (можно поставить EV-зарядку), не обязателен.

## 2. Источники и обход блокировок
- **idealista.pt** — DataDome. Брать только **headful Chrome на :0** + persistent profile (`.idealista-profile`) + **новая вкладка на каждый URL** (иначе detached-frame). Капчу решаешь 1 раз. НИКОГДА не парсить idealista из субагентов/WebFetch — словишь бан.
- **Поиск** новых: субагенты через **WebSearch** (idealista из поиска видно без захода на сайт) + порталы imovirtual, casa.sapo, supercasa, green-acres, idealista-альтернативы, агентские сайты, OLX/соцсети. Их страницы субагенты открывают свободно.
- Бот-защищённые не-idealista сайты (iad и т.п.) — тоже **headful**.

## 3. Конвейер (порядок скриптов)
1. `interactive2.cjs` — скрейп idealista-empreendimentos (таблица юнитов, цена из `<strong>`).
2. `search.cjs` — скрейп idealista-поиска (URL с shape), пагинация.
3. `phase1.cjs` / `allimg.cjs` — детали + полные галереи idealista.
4. `nonide_*` / `n2_*` — не-idealista: фото, координаты, описания.
5. `dedup3.py` — дедуп (перцептивный хэш фото + цена + описание).
6. montage → vision-субагенты — дизайн-рейтинг, тип, render/фото, выбор 4 фото; `coordfix`/ocean.
7. `score_rank.py` → `build_final2.py` → `gen_html_final.py` — оценка, превью, HTML.

## 4. ПРАВИЛА ОБНОВЛЕНИЯ
**Каденс:** полный прогон раз в ~2-4 недели; быстрая проверка ссылок — еженедельно.

### 4.1 Даты и статус жизненного цикла (НОВОЕ)
- `added.json` = реестр `{id: "YYYY-MM-DD"}`, **персистентный**. Колонка «Добавл.» в таблице. Дату первого появления существующим **не менять**.
- Завести также (рекомендация codex): `last_checked` (когда последний раз сверяли поля со страницей), `last_seen` (когда страница последний раз была жива), `last_changed`, и `status`: `active | removed | uncertain | duplicate | out_of_scope` + причина.
- **Не удалять бесследно**: снятые/неподтверждённые → статус + дата, исключать из топов, переносить в `archive.json` с последним известным состоянием (а не стирать).

### 4.2 Добавление нового объекта (полный пайплайн «по той же схеме»)
1. Проверить критерии (§1) и что это реально **T3** (не диапазон T2/T3 — иначе пометить ⚠).
2. Скачать **фото** (≥5; idealista — полную галерею; не-idealista — до 12 с источника/альт-источника).
3. **Координаты** (из «mapa» idealista или адреса/застройки; пометить точные/прибл.).
4. **Описание** (для реюза/классификации).
5. **Дизайн-рейтинг** (vision по montage; пусто — ок, если кухня/санузлы современные).
6. **Год**: со страницы; если нет — оценка vision+codex, помечать **≈**.
7. **Дедуп** против всей базы (§4.3).
8. Поставить `added` = текущая дата. Пересчитать score/rank.

### 4.3 Дедуп (обязательно при каждом обновлении)
- Основной сигнал — **общие фото** (perceptual dHash, ≥2-3 совпадения = один объект/дом).
- Рендеры одного проекта хэш не ловит → доп. сверка по **цене+зоне+площади**, затем vision/codex подтверждает.
- «Один дом, разные квартиры» = дубликат: оставить 1 представителя (idealista/больше фото/дешевле), цены/источники — в `dupnote`.
- Codex помогает текстовой сверкой (название застройки/адрес/цена).

### 4.4 Верификация (чтобы не вводить в заблуждение)
- **Битые ссылки**: открывать каждую (не только HTTP-код — homepage/search отдают 200!). Снято/404/не-T3 → убрать; T3-цена не подтверждена (диапазон/не построено) → ⚠.
- idealista — выборочная headful-перепроверка T3+цены.
- Локация: маркер не должен указывать на офис агентства (частая ошибка) — брать адрес объекта.

### 4.5 Производные поля
- **Расстояние до моря** — считать по координатам (хаверсин до ближайшего пляжа), НЕ по зоне.
- **Новостройка** — строго по описанию/состоянию (не просто «год ≥2022»).
- Площадь — разбивать на **общую/жилую/террасу**.


### 4.6 Поиск обновлений + уроки (round 2026-05-30)
- **Поиск везде** (не-idealista): fan-out субагентов по PT-порталам, международным, агентским сайтам, соцсетям/классифайдам. Рынок нового A/B ≤1M T3 в этой зоне **тонкий** — большинство находок уже в базе / >1M / energy C / до 2015 / сняты.
- **Год сдачи проверять обязательно**: codex-web-check поймал «новостройку», у которой реально **completion Oct 2027** (нарушает «не 2027+ off-plan») — дропнули.
- **Photo-hash слаб для рендеров/новостроек** (один проект — разные рендеры): доп. текстовая сверка (название проекта/адрес/застройщик).
- **E&V/некоторые сайты** отдают только стоковые фото агента и нестабильные expose-ссылки → не класть в публичную базу (только в приватный backlog).
- **Off-plan флажить**: проверять лицензию, срок сдачи, график оплаты, банк-гарантию; площадь общая vs жилая (терраса в общей).
- **Свежие первыми**: дефолтная сортировка списка = по дате добавления (новые сверху, поле `addseq` = порядок в `added.json`).

## 5. Score / Ranking
- **Score** (0-100) = взвеш. сумма **перцентильно** нормированных параметров (устойчиво к выбросам): цена↓, дизайн↑, год↑, близость к морю↑, размер↑, новостройка↑. Веса — живые слайдеры.
- **Rank** = `score − λ·ndom`, где ndom = сколько объектов доминируют по Парето {цена↓,дизайн↑,год↑,море↓,размер↑}. ⭐ = Парето-оптимум (ни один не лучше по всем осям). Топ-подборки = объекты, проходящие критерий, с лучшим rank.

## 6. Известные оговорки (от ревью codex)
- Дизайн-оценка субъективна (по фото; рендеры могут льстить).
- Часть координат — уровень улицы/зоны (помечено 📍≈), не точные.
- Годы с **≈** — оценка (фото/описание), не из документов.
- Цены — запрашиваемые, на дату скрейпа; диапазонные у застроек помечены ⚠.

## 8. Улучшения процесса (ревью codex — внедрить)
1. **Стабильная идентичность объекта** — `canonical_id`, `source_ids[]`, `urls[]`, `development_id`, `unit_id`. Чтобы база переживала переобъявления, смену агентства, новые фото и новые URL без потери связи и без задвоений.
2. **Учёт development vs unit** — разные T3-юниты одной застройки не сливать как дубли и не задваивать; хранить связь unit→development.
3. **Changelog `history.json`** (append-only) — diff по объекту при каждом прогоне: old/new цена, статус, URL, ключевые поля, timestamp, кто/что подтвердило. Защита от «silent drift» (объект остался, а цена/энергокласс/тип уже изменились).
4. **Еженедельный health-check** (лёгкий) — открыть каждую ссылку, обновить `last_checked/last_seen/status/price_available`, БЕЗ полной перегенерации базы. Полный прогон — реже.
5. **Validation report перед сборкой HTML** — список: новые / изменённые / снятые / подозрительные дубли / объекты с просроченной проверкой (старше N дней). Только после ревью — ребилд.
6. **Провенанс по полю** — у price/typology/energy/year/parking/coords хранить: подтверждено-страницей vs оценено-vision/codex, source URL, дата проверки, confidence. (Сейчас частично: ≈ у года, 📍≈ у координат, ⚠ у диапазонных цен.)
7. **Усилить дедуп-ключи** — кроме фото/цены добавить нормализованные: название проекта, гео-бакет/адрес, площадь, этаж, референс агентства, ценовой диапазон (ловит переопубликованные объявления с новыми фото).
8. **Верификация фиксирует поля**, а не только «страница открылась 200» (homepage/search тоже 200): сверять price, typology=T3, energy, year/new-build, parking, coords.

## 10. Публикация / обновление сайта (ОБЯЗАТЕЛЬНО после каждого обновления базы)
- **Сайт:** https://alexkirkouski.github.io/algarve-t3/ · **репо:** github.com/AlexKirkouski/algarve-t3 (public, GitHub Pages, ветка `main`, папка `/`, `.nojekyll`).
- **Правило:** после ЛЮБОГО изменения базы (добавление/дедуп/верификация/правки полей) — **переопубликовать**, иначе по ссылке останется старое. Колонка «Добавл.» показывает, что нового с прошлой публикации.
- **Одной командой:** `bash publish.sh` — пересобирает HTML из данных → ужимает новые фото в `deploy/` (инкрементально, ~1000px) → `commit` + `push`. Сайт обновляется по той же ссылке через ~1 минуту.
- **Фото:** в `deploy/images` ужаты до ~1000px (оригиналы — в `images/`, 402 МБ, в репо НЕ идут). downscale инкрементальный; если у объекта **заменили** фото — удалить `deploy/images/<id>` (или `FORCE=1 bash publish.sh`), чтобы пере-ужать.
- **Размер репо** ~164 МБ (в лимитах GitHub Pages, <1 ГБ). Репо публичный (нужно для бесплатного Pages); данные — открытые объявления.
- **Проверка после публикации:** дождаться статуса Pages `built` и открыть ссылку (HTTP 200 у `index.html` и у sample-картинки).
